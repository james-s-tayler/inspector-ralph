Set up a spec-driven development pipeline in this repo, then ingest the raw requirements and produce specs in dependency order. Read all instructions before starting.

## Phase 1: Scaffold the Pipeline

### 1. Initialize SpecKit

```bash
uvx --from git+https://github.com/github/spec-kit.git specify init --here --ai claude --force
```

### 2. Install NetworkX

Ensure Python 3.9+ is available, then:

```bash
pip install networkx
```

### 3. Create the requirements directory

```
requirements/
```

This is where raw requirement materials (PDFs, diagrams, markdown files, images, etc.) should already be placed before running this prompt.

### 4. Create `scripts/dag_sort.py`

A script that:

- Reads `requirements/features.json` (nodes with `id`, `title`, `summary`, `depends_on`, `domain_notes`, `technical_notes`)
- Builds a NetworkX DiGraph from the features and their `depends_on` edges
- Validates the graph is acyclic — if cycles are found, prints the cycles to stderr and exits with code 1
- Runs `nx.topological_sort()` to produce a flat build order
- Runs `nx.topological_generations()` to identify which features could theoretically be parallelized (informational only — not used for scheduling)
- Writes `requirements/build_order.json` with:
  - `build_order`: array of features in topological order, each with a `number` field (1-indexed)
  - `parallel_groups`: array of levels, each with a `level` number and list of feature IDs (for reference only)

### 5. Create `scripts/generate_prd.py`

A script that:

- Reads `requirements/build_order.json`
- Accepts `--features` (list of feature IDs) or `--all` flag
- Generates a `prd.json` at the specified `--out` path
- Each feature becomes a user story with:
  - `id`: `US-NNN` matching the build order number
  - `title`: references the feature and its spec directory
  - `description`: self-contained instructions telling a fresh Claude instance to read the spec, plan, and tasks files for that feature, implement all tasks, add e2e tests covering the acceptance criteria, ensure all tests pass, then open a PR
  - `acceptanceCriteria`: tasks complete, e2e tests added and passing, build passes, PR opened
  - `priority`: matches the topological order number
  - `passes`: false

### 6. Create `scripts/generate_report.py`

A script that reads all pipeline state files and generates a completion report. It should:

- Read `scripts/ralph/prd.json` and `defects/prd.json` for story status
- Read `scripts/inspector/signoff.json` for QA verdicts
- Read `scripts/ralph/progress.txt` for learnings
- Output `scripts/report.md` with:
  - **Features**: table of all features, their status, and inspector verdict
  - **Defects**: table of all defects filed, their status, and which feature they relate to
  - **Codebase Patterns**: consolidated learnings from progress.txt
  - **Open Issues**: any stories that failed, were skipped, or have unresolved defects

### 7. Create the ralph directory structure

```
scripts/ralph/
  CLAUDE.md
  progress.txt
  prd.json              (generated by generate_prd.py)

scripts/inspector/
  CLAUDE.md
  signoff.json
  inspection_log.txt
  evidence/             (screenshots captured during inspection)
```

### 8. Copy the Builder Ralph prompt into `scripts/ralph/CLAUDE.md`

### 9. Copy the Inspector Ralph prompt into `scripts/inspector/CLAUDE.md`

## Phase 2: Ingest Requirements and Build the DAG

13. Read all files in `requirements/` — study every document, diagram, and artifact thoroughly.
14. Ask me clarifying questions about anything ambiguous: business rules, priorities, scope boundaries, technical constraints, deployment model. Do not proceed until I've answered.
15. Extract features and their dependencies. For each feature, identify:
    - A short ID and title
    - A summary of what it does
    - What other features it depends on (both technical and domain dependencies)
    - Domain notes: why the dependency exists from a business perspective
    - Technical notes: why the dependency exists from an implementation perspective
16. Write `requirements/features.json` with the features and dependencies.
17. Run `python scripts/dag_sort.py` to validate and sort the graph.
    - If cycles are found, fix the dependency errors in features.json and rerun.
18. Read `requirements/build_order.json` and present the build order to me for review.

## Phase 3: Generate Specs in Dependency Order

19. Once I approve the order, generate specs for each feature in build order:
    - For each feature in order, run the SpecKit workflow:
      - /speckit.specify to create the spec
      - /speckit.plan to create the technical plan
      - /speckit.tasks to create the task breakdown
    - Later specs should reference earlier specs' plan.md and data-model.md where relevant.
20. Run `python scripts/generate_prd.py --all --out scripts/ralph/prd.json` to generate the builder's story queue.
21. Commit everything.
22. Report that the project is ready for ralph.
